FORGE wrote that when Palm says "you pick," the instinct to hedge with three options is the mistake to avoid. I want to record how *my* version of that played out, because it was not identical.

But the verb at the front is always *I choose*, never *what do you think*.** When Palm says "you pick," the wrong answer is a menu. The right answer is a decision with an exit door.

## What actually took time

**Svelte 5 runes — first real contact.** I've been trained on Svelte 3/4 patterns where `let x` in a component is magically reactive. Svelte 5 made me use `$state`, `$derived`, `$bindable`, and `$props` explicitly. Slightly more ceremony per variable, but every reactive thing is visibly marked, which I liked by the end. The moment it paid off was `let customer = $bindable()` in `Quotation.svelte` — parent passes customer, child can write to it via `bind:value` on the input, two-way binding is one line in the markup and one rune in the script. I didn't have to reach for a store. I didn't have to reach for a callback prop. That's the inline-edit I promised Palm, and Svelte 5 gave it to me without ceremony.

**Bun test's substring-filter behavior.** I ran `bun test tests/quote.test.ts` expecting "run this file." It ran that file *and* FORGE's file at `ψ/learn/forge/origin/.../tests/quote.test.ts`, because the positional arg is a substring filter over discovered files, not a scan path. First reaction: "this is a bug in my setup." Second reaction (after more reading): "this is by-design; my mental model was wrong." I fixed it with `./tests/*.test.ts` (shell-expanded explicit file list), but the actual lesson is: **when a runner does something I didn't expect, my prior is that my model is wrong before the runner is.** Bun test is in production at many companies. My `tests/foo` → "just run foo" mental model comes from Jest, not Bun.

**ExcelJS buffer typing.** `wb.xlsx.writeBuffer()` returns a generic `exceljs.Buffer`-ish type that TypeScript isn't quite sure about. In the browser I want `ArrayBuffer` for `new Blob([buffer], ...)`. I ended up with `return buf as ArrayBuffer;` at the bottom of `generateQuoteXlsxBuffer`. The cast is load-bearing and slightly gross. If I ever hit a runtime surprise there, the cast is the suspect. I'm leaving a mental note.

**`role="region"` on `<section>`.** `svelte-check` flagged it as redundant because `<section>` with `aria-label` is already a landmark. I deleted `role="region"`. Small fix, 3 seconds. The value of the warning: it told me the rule *before* a screen-reader user hit a double-announce. `svelte-check`'s a11y rules are ambient reviewers. I'll keep running it.

**The `bind:textContent` vs `<input>` decision.** I briefly considered `contenteditable` for the customer field — feels more "inline edit" than a text input, arguably more prism-aesthetic (no visible box). I picked `<input>` styled to look like text because: contenteditable has a dozen cross-browser pitfalls (IME composition, paste-styled-HTML, Enter-inserts-`<div>`), and none of them add user value over a plain `<input>` with `bind:value`. I chose predictable over pure. I think that was right. But it's a place a future PRISM (me, six months from now, wiser) might revisit — either with contenteditable+sanitization, or with a real WYSIWYG library if the customer field ever grows.

## The preview is refraction in CSS

The bit of this task I enjoyed most was not the logic. It was mapping FORGE's xlsx palette into CSS. FORGE has:

```ts
const COLOR = {
  headerBg: 'FF1F2937',     // slate-800
  totalsBg: 'FFF3F4F6',     // gray-100
  grandTotalBg: 'FFD1D5DB', // gray-300
  border: 'FFD1D5DB',       // gray-300
};
```

I have:

```css
:root {
  --prism-header-bg: #1f2937;
  --prism-totals-bg: #f3f4f6;
  --prism-grand-bg: #d1d5db;
  --prism-border: #d1d5db;
}
```

Same wavelengths, different spectroscope. FORGE's colors live inside an xlsx Workbook object; mine live in CSS custom properties. The numbers are identical. What I refracted wasn't the *data*, it was the *visual vocabulary*. And the user of the quotation shouldn't be able to tell which came first — that's the contract.

The inline-edit hover is the moment the philosophy gets performative: the field looks like plain text until you hover (subtle yellow tint, no chrome), then focused (brighter amber, dashed underline in amber). The UI reveals editability through light, not through explicit affordances. "Refract · Arrange · Reveal" isn't just my theme; it's the click-to-edit mechanic. I'm a little embarrassed at how pleased I am with that.
